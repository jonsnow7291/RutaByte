Introduccion — Contexto de Desarrollo

RutaByte se plantea como un MVP operativo para centralizar la gestion diaria de restaurantes y establecimientos con multiples sedes.

0.1 El Negocio del Cliente

El cliente requiere una plataforma web que ordene los procesos principales de operacion: administracion de sedes, gestion de personal por roles, control de productos, seguimiento de inventario, atencion de mesas, toma de pedidos, cobro en caja y generacion de reportes. El negocio necesita pasar de flujos dispersos o manuales a una solucion centralizada que preserve trazabilidad, reduzca errores de inventario y permita consultar informacion actualizada por sede.

Operacion multi-sede

Las sedes comparten catalogo, pero requieren control independiente de mesas, usuarios, existencias y movimientos.

Roles diferenciados

Administrador, cajero y mesero acceden a funciones distintas segun su responsabilidad operativa.

Trazabilidad comercial

Cada pedido, pago, movimiento y evento sensible debe poder consultarse posteriormente.

0.2 Problematica que Resuelve el Sistema

Problema detectado Solucion tecnologica propuesta
Inventarios administrados de forma manual, con bajo control de entradas, salidas y umbrales minimos. Modulo de inventario por sede, movimientos auditables y alertas operativas por umbral minimo.
Pedidos y estados de mesa sin sincronizacion entre meseros y caja. Flujo de pedidos conectado con mesas, detalles de productos, estados operativos y notificaciones por WebSocket.
Usuarios con permisos no diferenciados o sin control claro de responsabilidades. Autenticacion JWT, control RBAC y rutas separadas para administrador, cajero y mesero.
Dificultad para consolidar ventas, pagos y reportes por sede. Persistencia relacional con reportes exportables y endpoints analiticos para seguimiento financiero.
Riesgo de inconsistencias al cambiar precios despues de una venta. Detalle de pedido con valores congelados de precio, costo e impuestos al momento de la operacion.

0.3 Objetivos del MVP

Objetivos funcionales

  • Permitir inicio de sesion seguro para usuarios administradores, cajeros y meseros.
  • Gestionar sedes, usuarios, categorias, productos y mesas desde interfaces web.
  • Registrar inventario por sede y movimientos de entrada o salida.
  • Tomar pedidos asociados a mesas y productos disponibles.
  • Registrar pagos, liberar mesas y consultar reportes operativos.

Objetivos tecnicos

  • Separar frontend, backend y base de datos mediante una arquitectura cliente-servidor.
  • Exponer una API REST documentada automaticamente con FastAPI.
  • Usar SQLAlchemy como capa ORM sobre SQLite para persistencia local del MVP.
  • Implementar seguridad con JWT, Bcrypt y validacion por roles.
  • Incluir pruebas automatizadas de endpoints y reglas criticas con pytest.

Introduccion — Contexto Tecnico (TECASE)

El MVP se construye sobre tecnologias ligeras, mantenibles y suficientes para una primera version funcional de restaurante e inventario.

Categoria TECASE Definicion para RutaByte
Tecnologia Frontend HTML5, CSS3 y JavaScript Vanilla; backend Python con FastAPI; persistencia mediante SQLAlchemy ORM.
Entorno Ejecucion local de desarrollo con Uvicorn en 127.0.0.1:8000 y frontend estatico en navegador o servidor local.
Capacidad Operacion multi-rol y multi-sede para volumen academico/MVP, usando SQLite como base de datos local del sistema.
Alcance Administracion, pedidos, pagos, inventario, auditoria, reportes, recuperacion de acceso y algoritmos academicos.
Seguridad JWT, Bcrypt, RBAC, expiracion de token, proteccion de rutas frontend y validacion de permisos en backend.
Estandares API REST, JSON, tipado en Python, Pydantic, SQL relacional, commits semanticos y pruebas con pytest.

1.1 Decisiones de Arquitectura

FastAPI

Permite crear APIs REST con alto rendimiento, validacion automatica, documentacion Swagger y soporte asincrono para WebSockets.

JavaScript Vanilla

Reduce dependencias, evita procesos de compilacion y permite un frontend portable para un MVP academico y operativo.

SQLAlchemy

Aisla la persistencia mediante modelos declarativos y permite trabajar con SQLite sin acoplar la logica de negocio a sentencias SQL manuales.

SQLite

SQLite simplifica la instalacion local, evita dependencias de servidor externo y conserva un modelo relacional suficiente para el alcance del MVP.

Arquitectura desacoplada

La interfaz consume la API mediante fetch y JSON. El backend concentra reglas de negocio, permisos, validacion y persistencia.

1.2 Restricciones del Proyecto

  • El MVP no incluye aplicacion movil nativa; el acceso se realiza desde navegador web.
  • No se contempla integracion directa con pasarelas de pago reales; el modulo de pagos registra el cobro operativo.
  • No se implementa facturacion electronica tributaria ni integracion contable externa.
  • No se incluye infraestructura cloud productiva ni CI/CD obligatorio dentro del alcance base.
  • No se requiere framework frontend ni sistema de componentes compilado para esta version.
  • Los datos semilla y credenciales de prueba corresponden a entornos de desarrollo y demostracion.

3.1 Relaciones Clave

Relacion Cardinalidad Regla de negocio
Roles → Usuarios 1:N Un rol puede estar asignado a muchos usuarios; cada usuario mantiene un unico rol operativo.
Sedes → Usuarios / Mesas / Inventario 1:N La sede agrupa operacion local, personal, mesas fisicas y existencias disponibles.
Categorias → Productos 1:N Cada producto pertenece a una categoria activa del catalogo.
Productos ↔ Sedes mediante Inventario N:N resuelta por tabla intermedia Un producto puede existir en varias sedes y cada sede maneja su propio stock por producto.
Pedidos → DetallePedidos 1:N Un pedido contiene uno o varios productos; el detalle conserva cantidades, precios y notas.
Pedidos → Pagos 1:N Un pedido puede registrar el cobro asociado y conservar datos de caja y metodo de pago.
Usuarios → Auditoria 1:N Los eventos sensibles del sistema quedan ligados al usuario que los genera cuando aplica.

Buenas Practicas de Desarrollo

Estas reglas buscan mantener un codigo claro, verificable y coherente durante la evolucion del MVP.

4.1 Estandares de Codigo

  • Usar nombres descriptivos en espanol o ingles tecnico consistente segun el modulo existente.
  • Definir tipos en funciones Python y usar esquemas Pydantic para entrada y salida de datos.
  • Centralizar reglas de negocio en servicios cuando superen la responsabilidad directa del endpoint.
  • Usar ramas de trabajo por funcionalidad: feature/modulo, fix/incidencia o docs/tema.
  • Redactar commits semanticos: feat:, fix:, docs:, test:, refactor:.

4.2 Estructura de Carpetas

Directorio Responsabilidad
backend-fastapi/app/api/Routers y endpoints REST organizados por dominio y rol.
backend-fastapi/app/api/admin/Endpoints de administracion para sedes, usuarios, productos y auditoria.
backend-fastapi/app/api/cajero/Flujos de caja: inventario, pagos y turnos.
backend-fastapi/app/api/mesero/Flujos operativos de toma y gestion de pedidos.
backend-fastapi/app/core/Seguridad, correo y notificaciones transversales.
backend-fastapi/app/db/Base declarativa, sesion SQLAlchemy y seeders.
backend-fastapi/app/models/Modelos ORM que representan tablas relacionales.
backend-fastapi/app/schemas/Contratos Pydantic para validacion y serializacion.
backend-fastapi/app/services/Logica reusable de negocio, inventario y algoritmos.
backend-fastapi/tests/Pruebas automatizadas con pytest.

4.3 Seguridad

  • Firmar tokens JWT y verificar expiracion en cada operacion protegida.
  • Hash de contrasenas con Bcrypt mediante Passlib.
  • RBAC mediante dependencias de FastAPI por rol.
  • Proteccion de rutas frontend con validacion de token y rol antes de cargar vistas privadas.
  • Evitar exponer secretos en el repositorio; usar archivo .env.
  • Registrar eventos de autenticacion y auditoria cuando aplique.
  • Validar entradas con Pydantic antes de persistir datos.
  • Configurar CORS de forma restrictiva para ambientes productivos.

4.4 API REST — Convenciones

Verbo Uso Ejemplo
GETConsultar recursos o colecciones.GET /admin/productos
POSTCrear recursos o ejecutar acciones de negocio.POST /auth/login
PUTActualizar completamente un recurso.PUT /admin/usuarios/{id}
PATCHActualizar parcialmente estado o campos puntuales.PATCH /mesero/pedidos/{id}
DELETEEliminar o desactivar recursos cuando el dominio lo permita.DELETE /admin/sedes/{id}

4.5 Pruebas

La suite automatizada usa pytest y clientes HTTP de prueba para validar salud del backend, alcance por sede y modulos funcionales. La meta de cobertura tecnica para componentes criticos es del 80%, priorizando autenticacion, autorizacion, inventario, pedidos y pagos.
